Clean Architecture(Robert C. Martin) 的核心思想是通过分层和依赖规则，实现软件系统的高内聚、低耦合，从而提升可维护性、可测试性和可扩展性。其关键在于将业务逻辑与外部技术细节（如数据库、框架、UI、Web 服务等）彻底分离，确保系统的核心——也就是业务规则——不依赖于任何外部组件。

这一架构强调“依赖倒置原则”：高层模块（如用例、业务逻辑）不应依赖低层模块（如数据库、网络），两者都应依赖于抽象。所有依赖关系必须指向系统内部，即从外层（技术细节层）向内层（业务核心层）流动，而内层对最外层一无所知。

Clean Architecture 通常将系统划分为多个同心圆层，从内到外依次是：Entities（实体，代表核心业务对象和规则）、Use Cases（用例，封装具体业务逻辑）、Interface Adapters（接口适配器，如控制器、数据转换器，用于连接内外）、Frameworks and Drivers（框架与驱动，包括数据库、Web 框架、UI 等）。越靠近中心，越稳定、越通用；越靠外层，越易变、越具体。

通过这种结构，业务逻辑可以独立于技术栈进行开发和测试，即使更换数据库、UI 框架或部署方式，也不会影响核心业务规则，从而显著提升系统的长期可维护性和适应变化的能力。

它到底想解决什么？

想象一下：你写了一个用户注册功能，直接在 Flask 路由里连数据库、校验邮箱、插入数据——代码跑得挺快，但很快问题来了：

  • 想换数据库？得改业务逻辑。
  • 想写单元测试？得先起个数据库。
  • 想加个命令行入口？得把 Web 逻辑拆出来。
  • 业务规则一变，整个文件都得动。

Clean Architecture 的核心思路很简单：把“业务逻辑”和“技术细节”分开，让前者不依赖后者。

怎么分？靠两件事：

  1. 分层：把代码按职责分成几层，越往里越“纯粹”，只关心业务；越往外越“具体”，处理数据库、HTTP、文件等。
  2. 依赖方向：内层不知道外层的存在，外层去实现内层定义的接口。

这样一来，换数据库、加 API、写测试，都只动外层，核心逻辑几乎不用改。

我们用一个极简的用户注册功能来演示。项目结构如下：

user_service/
├── main.py
├── entities/
│   └── user_repository.py
├── use_cases/
│   └── register_user.py
├── adapters/
│   ├── sqlite_user_repository.py
│   └── in_memory_user_repository.py
└── tests/
    └── test_register_user.py

第一步：定义“我需要什么”（内核）

业务逻辑说：“我需要一个能存用户的地方。”
但它不关心是存到 SQLite、PostgreSQL，还是内存里。所以，我们先定义一个接口：

# entities/user_repository.py
class UserRepository:
    def save(self, name: str, email: str):
        raise NotImplementedError

然后写注册逻辑，只依赖这个接口：

# use_cases/register_user.py
from entities.user_repository import UserRepository

class RegisterUser:
    def __init__(self, user_repo: UserRepository):
        self.user_repo = user_repo

    def execute(self, name: str, email: str) -> str:
        if "@" not in email:
            raise ValueError("邮箱格式错误")
        self.user_repo.save(name, email)
        return "注册成功！"

注意：这段代码里没有任何 import sqlite3、import flask，也没有任何具体技术。它只做一件事：描述业务规则。

第二步：实现“怎么存”（外层）

现在，我们提供两种实现：

# adapters/sqlite_user_repository.py
import sqlite3
from entities.user_repository import UserRepository

class SqliteUserRepository(UserRepository):
    def save(self, name: str, email: str):
        conn = sqlite3.connect("users.db")
        conn.execute("INSERT INTO users (name, email) VALUES (?, ?)", (name, email))
        conn.commit()
        conn.close()
# adapters/in_memory_user_repository.py
from entities.user_repository import UserRepository

class InMemoryUserRepository(UserRepository):
    def __init__(self):
        self.users = []

    def save(self, name: str, email: str):
        self.users.append({"name": name, "email": email})

这两个类都实现了同一个接口，但一个写数据库，一个只存内存。业务逻辑完全不用知道区别。

第三步：对象装配

在程序入口，我们决定用哪个实现：

# main.py
from use_cases.register_user import RegisterUser
from adapters.sqlite_user_repository import SqliteUserRepository

if __name__ == "__main__":
    repo = SqliteUserRepository()
    use_case = RegisterUser(repo)
    print(use_case.execute("小明", "xiaoming@example.com"))

如果哪天想换数据库，只需要改这一行：repo = PostgresUserRepository()，其他地方不动。

第四步：测试变得很简单

因为业务逻辑不依赖数据库，测试可以直接用内存实现：

# tests/test_register_user.py
from use_cases.register_user import RegisterUser
from adapters.in_memory_user_repository import InMemoryUserRepository

def test_invalid_email():
    repo = InMemoryUserRepository()
    use_case = RegisterUser(repo)
    try:
        use_case.execute("小明", "bad-email")
        assert False
    except ValueError:
        pass  # 预期行为

不需要 mock 数据库连接，也不用清理测试数据，跑得快，结果稳。

依赖关系长什么样？

main.py
   │
   ▼
use_cases/ ────依赖───▶ entities/
   ▲                    ▲
   │                    │
adapters/ ───实现───────┘
  • use_cases 只知道 entities 里的接口；
  • adapters 实现这些接口；
  • main.py 负责把它们“拼”在一起；
  • 内层代码从不 import 外层。

这就是所谓的“依赖向内”——不是规定，而是为了隔离变化。

小结

一开始觉得这种结构“太重”，一个小功能搞这么多文件。但后来发现：

  • 当项目变大，改需求、加功能时，知道哪些代码能动、哪些不能动，比什么都重要；
  • 测试写起来轻松很多，尤其是回归测试；
  • 团队协作时，有人改数据库、有人写 API，只要接口不变，互不干扰。

当然，不是所有项目都需要这么分。如果你写个一次性脚本，直接写也没问题。但当你希望代码“活得久一点”，这种分离会带来长期收益。

Clean Architecture 不是银弹，也不是必须遵守的教条。
它只是提供了一种思路：让业务逻辑独立于技术细节。
目录结构、分层命名，都是为了服务于这个目标。

可以从一个小功能开始尝试——比如把数据库操作抽成一个接口，业务逻辑只依赖它。
不用一步到位，慢慢体会它带来的好处就好。
