Go 内存管理机制解析
2026-04-21 23:50:00

概览

基于 Go 源码 src/runtime/ 的深度分析，所有结构体定义和常量均来自实际源码。

当你写下 make([]byte, 100) 的时候，Go 是怎么给你一块内存的？

最直接的做法是问操作系统 —— 调用 mmap 系统调用。但一次系统调用的开销是几百纳秒到几微秒，而 Go 程序可能每秒分配几百万次。如果每次都问 OS，程序就别干正事了。

Go 的解决思路和现实中的物流体系很像：不要每次需要一颗螺丝就跑去工厂，而是在桌上放一盒、部门放一柜、仓库放一批、工厂按吨生产。这就是 Go 内存分配器的核心设计：多级缓存 + 批量获取 + 无锁快速路径。它源自 Google 的 tcmalloc（Thread-Caching Malloc），但经过多年演化已经是完全不同的设计（src/runtime/malloc.go）。

具体来说，Go 的内存分配有四个层级：

层级 类比 实际结构 特点
1 你桌上的工具箱 mcache（Per-P） 只有你自己用，拿东西不用打招呼
2 部门的工具间 mcentral（Per-Size） 部门共用，需要登记（加锁），但一次换一整箱
3 公司中央仓库 mheap（全局唯一） 全公司共用，管理所有库存和场地
4 外部供应商 OS（mmap） 最慢，但每次大批量进货

绝大部分分配在第 1 层就完成了 —— 从 mcache 的缓存 span 中通过一条 CPU 指令找到空闲 slot，整个过程无锁、无竞争，耗时个位数纳秒。只有当本地缓存耗尽时，才会逐级向上申请。每向上一层都更慢，但也更”大方”—— 以更大的粒度批量补货，摊销了锁和系统调用的开销。

[!tip] 核心思想
每一层都在做同一件事：用”批量获取 + 本地缓存”来避免频繁访问更慢的下一层。这和 CPU 的 L1/L2/L3 缓存是完全一样的思想。

有了全景之后，我们从最底层开始：Go 向操作系统拿到的内存，是怎么组织起来的？又是怎么切成程序能用的对象的？

Arena、Size Class 与 mspan

Go 的堆可能有几个 GB，不可能给每个字节单独记账。就像管理一个巨大的仓库，合理的做法是先把仓库划成若干区域，每个区域配一本管理簿。Go 把这些区域称为 Arena。

平台 Arena 大小 原因
64-bit 非 Windows 64 MB 足够大以减少 arena 数量，对齐方便位运算
Windows / 32-bit 4 MB 虚拟内存管理粒度不同
WebAssembly 512 KB 内存受限环境

每个 arena 的起始地址对齐到自身大小，通过简单的位运算就能从任意地址算出它属于哪个 arena（src/runtime/malloc.go）。堆需要更多空间时，mheap 会向 OS 申请新的虚拟内存并注册新 arena —— arena 的数量是动态增长的，不是一开始就固定的。

Arena 里面有什么？ Go 为每个 Arena 维护一个 heapArena 结构体，就是这块区域的”管理簿”（src/runtime/mheap.go）：

type heapArena struct {
    spans     [pagesPerArena]*mspan          // 页 → 管理单元的映射（8192 条目）
    pageInUse [pagesPerArena / 8]uint8       // 位图：哪些页正在使用
    pageMarks [pagesPerArena / 8]uint8       // 位图：哪些页有存活对象
    zeroedBase uintptr                       // 零初始化水位线
}

这里先注意两件事。第一，spans 数组把 arena 内的每一页（8KB）映射到一个”管理单元”——也就是 mspan，我们马上会详细介绍它。现在只需要知道，mspan 是 Go 管理内存的核心单元，而 heapArena 通过 spans 数组记录了”这一页属于哪个 mspan”。第二，zeroedBase 是个巧妙的优化。OS 通过 mmap 给的新页本来就是零，而页分配器用地址有序的 first-fit 策略，新页地址总是递增的。zeroedBase 作为水位线，标记”此处以下的内存保证已清零”，省掉了大量不必要的 memset。

mheap 怎么管理这些 Arena？ 所有 arena 的管理簿汇总在全局唯一的 mheap 中，通过一个二级查找表组织：

// src/runtime/mheap.go
type mheap struct {
    arenas [1 << arenaL1Bits]*[1 << arenaL2Bits]*heapArena
    // ...
}

给定任意一个堆上的指针，通过整数除法算出 arena 索引，查表拿到 heapArena，再按页偏移查 spans 数组 —— 整条链路全是数组索引，O(1) 完成，没有任何搜索。这个能力对 GC 至关重要：GC 需要频繁判断一个指针指向哪个管理单元。

[!note] 关于二级表
大多数 64 位平台上 L1 位数 = 0，arenas 退化为一维平面数组，编译器会优化掉 L1 索引。二级结构只是为了兼容不同平台。

Arena 解决了”怎么组织原始内存”的问题。但 Arena 只是一大片连续的页，程序需要的是各种大小的对象。下一个问题自然是：怎么把这些页切成程序能用的对象？ 在切之前，我们先要回答一个前置问题：切成多大的块？

Size Class：把”任意大小”变成”有限尺码”。 如果程序请求多少字节就给多少字节（像 C 的 malloc），很快就会遇到碎片问题：先分配 100B，再分配 200B，释放 100B 后请求 150B —— 那个 100B 的空洞放不下 150B，内存就像被啃得七零八落的奶酪。Go 的解决方案是把所有对象归入有限的”尺码”，就像买衣服只有 S/M/L/XL。你要 100 字节？给你 112 字节。浪费一点，但同尺码的对象可以完美复用对方的空间，碎片消失了。

Go 定义了 68 个 size class（class 0 保留，1-67 实际使用），涵盖 8B 到 32KB（src/internal/runtime/gc/sizeclasses.go）。设计约束是 **每个 size class 的内部浪费不超过 12.5%**。

const (
    NumSizeClasses = 68
    MaxSmallSize   = 32768   // 32 KB，小对象上限
    TinySize       = 16      // Tiny 分配器阈值
    PageShift      = 13      // pageSize = 8KB = 1 << 13
)

从请求的 size 到 class 的查找是 O(1) 查表，没有循环：小于 1016 字节用 8B 步进查表，更大的用 128B 步进。

Span Class：再区分含不含指针。 尺码定好之后，Go 还做了一个额外的区分：这个尺码的对象含不含指针？

type spanClass uint8   // sizeclass<<1 | noscan
// 68 size class × 2 = 136 个 span class

含指针的对象在 GC 时需要扫描（追踪它引用了谁），不含指针的可以直接跳过。把它们分开管理，GC 就不用浪费时间扫描一堆 []byte。68 个 size class × 2 = 136 个 span class —— 这个数字会反复出现：mcache 有 136 个缓存槽，mheap 有 136 个 mcentral，每个 span class 独立管理。

现在我们有了尺码体系。回到之前的问题：怎么把 Arena 里的连续页，按某个尺码切成等大的格子？ 这正是 mspan 做的事。

mspan：Arena 上的”货架”。 简单来说，mspan 就是从 Arena 中拿出若干连续页，按某个 size class 切成等大 slot 的”货架”。它把前面的概念连接在一起：Arena 提供原始页，Size Class 定义尺码，mspan 把两者结合 —— 在一块连续的页上划出固定大小的格子，然后追踪哪些格子已用、哪些空闲。

// src/runtime/mheap.go:422-516
type mspan struct {
    startAddr  uintptr        // 起始地址（在 Arena 中的位置）
    npages     uintptr        // 占多少页

    freeindex  uint16         // 搜索空闲 slot 的起点
    nelems     uint16         // slot 总数
    elemsize   uintptr        // 每个 slot 的大小（由 size class 决定）

    allocBits  *gcBits        // 分配位图：1=已用, 0=空闲
    gcmarkBits *gcBits        // GC 标记位图：1=存活
    allocCache uint64         // allocBits 补码缓存（CTZ 加速）

    spanclass  spanClass      // 所属 span class
    sweepgen   uint32         // sweep 代
    allocCount uint16         // 已分配对象数
}

startAddr + npages 定位它在 Arena 中的位置，spanclass 说明它的尺码和 scan/noscan 类别，allocBits 追踪每个 slot 的使用状态。一个完整的信息闭环。

CTZ 快速分配。 知道了 mspan 的结构，下一个问题是：怎么在上面快速找到一个空闲 slot？Go 的做法非常巧妙。allocCache 存储 allocBits 的补码（1 = 空闲），分配时用 CPU 的 CTZ（Count Trailing Zeros） 指令直接定位第一个空闲位：

// src/runtime/malloc.go:969-985
func nextFreeFast(s *mspan) gclinkptr {
    theBit := sys.TrailingZeros64(s.allocCache)  // 一条 CPU 指令
    if theBit < 64 {
        result := s.freeindex + uint16(theBit)
        if result < s.nelems {
            s.allocCache >>= uint(theBit + 1)
            s.freeindex = result + 1
            s.allocCount++
            return gclinkptr(uintptr(result)*s.elemsize + s.base())
        }
    }
    return 0
}

一条 CTZ 指令在 64 个 slot 中瞬间找到第一个空闲的，无锁、无分支预测失败。这就是 Go 小对象分配做到个位数纳秒的原因。

位图交换：优雅的批量回收。 分配的问题解决了，回收呢？mspan 有两个位图：allocBits（谁被分配了）和 gcmarkBits（GC 标记谁还活着）。Sweep 时不需要逐个释放对象，只需要把 gcmarkBits（标记了谁还活着）变成新的 allocBits（谁还在用）—— 未标记的对象”自动”变成了空闲。

[!tip] 关键洞察
交换两个位图指针就完成了所有回收，比 C 的逐对象 free() 高效得多。

mspan 的生命周期。 mspan 不是静止的，它在系统中持续流转：mheap 在 Arena 中找到连续空闲页，创建 mspan，放入 mcentral 的 partial 列表。mcache 从 mcentral 获取 mspan，Goroutine 在上面分配对象。当 mspan 上所有 slot 用完，mcache 把它还给 mcentral 的 full 列表。GC sweep 后如果有对象死亡，mspan 回到 partial 列表继续服务；如果所有对象都死了，整个 mspan 的页归还 mheap。

这里出现了三个名字：mcache、mcentral、mheap —— 正是开篇全景图中的三级缓存。下一节我们看它们怎么协作，以及所有分配的统一入口 mallocgc。

分配：三级缓存与 mallocgc

上一节搭好了”舞台”——Arena 提供场地，Size Class 定义规格，mspan 切好格子。这一节看谁持有这些格子、怎么分发、怎么回收 —— 也就是三级缓存的协作机制，以及将一切串联起来的入口函数 mallocgc。

mcache：Per-P 无锁缓存。 分配的第一站是 mcache，它绑定在每个 P（Processor）上（src/runtime/mcache.go）：

type mcache struct {
    tiny       uintptr                       // Tiny 分配器
    tinyoffset uintptr
    tinyAllocs uintptr
    alloc      [numSpanClasses]*mspan        // 136 个 span 指针
}

为什么无锁？Go 的 GMP 模型保证同一时刻一个 P 上只有一个 Goroutine 在跑。mcache 绑定在 P 上，只有当前 Goroutine 会访问它，天然没有竞争。

分配 128B noscan 对象的过程：算出 spanClass = 21，从 alloc[21] 取出 mspan，调用 nextFreeFast 用 CTZ 找空闲 slot。99% 的分配到这里就结束了。

mcache 还有一个针对极小对象的额外优化 —— Tiny 分配器。对于 <16B 且不含指针的对象（小字符串、bool 等），多个对象合并到同一个 16B block 中。JSON 基准测试中这减少了约 12% 的分配次数和 20% 的堆内存。约束是仅限 noscan，否则一个对象存活会导致整个 block 无法回收。

但 mcache 的容量是有限的 —— 每个 span class 只缓存一个 mspan。当这个 mspan 的 slot 用完了怎么办？

mcentral：Per-SpanClass 中心缓存。 mspan 用完时，mcache 需要从 mcentral 换一个新的。全局共 136 个 mcentral，每个负责同一 span class 的所有 mspan（src/runtime/mcentral.go）：

type mcentral struct {
    spanclass spanClass
    partial   [2]spanSet    // 有空闲 slot 的 span
    full      [2]spanSet    // 没有空闲 slot 的 span
}

partial 和 full 各有 2 组，这是 sweepgen 双缓冲设计：每个 GC 周期 sweepgen 加 2，两组角色互换（swept ↔ unswept），新周期只需切换索引而无需清空列表。

mcache 来要 mspan 时，mcentral 按优先级搜索（src/runtime/mcentral.go）：先找已清扫的 partial span（最快），再找未清扫的 partial span（先 sweep 再给，限 100 个），再试 full unswept span（sweep 后可能腾出空间），最后才问 mheap 要新页。

[!important] 批量交换
mcentral 的操作粒度是整个 span，不是单个对象。mcache 一次换走一整个 span（可能有几十上百个空闲 slot），后续分配都在本地完成。这就是摊销锁开销的关键。

那如果 mcentral 也没有可用的 span 呢？请求继续上升到最后一层。

mheap：全局堆管理器。 mheap 是整个体系的最终后盾，全局唯一的堆管理器，管理着所有 arena、所有 mcentral、以及页分配器（src/runtime/mheap.go）：

var mheap_ mheap

type mheap struct {
    lock      mutex                                              // 堆锁
    pages     pageAlloc                                          // 页分配器
    arenas    [1 << arenaL1Bits]*[1 << arenaL2Bits]*heapArena    // Arena 映射
    central   [numSpanClasses]struct {                           // 136 个 mcentral
        mcentral mcentral
        pad      [...]byte                                      // cache-line 对齐
    }
    allspans  []*mspan                                           // 所有 span（只追加）
    sweepgen  uint32                                             // 当前 sweep 代

    spanalloc  fixalloc     // 分配 mspan 结构体本身
    cachealloc fixalloc     // 分配 mcache 结构体本身
}

看这个结构体，前面所有概念都汇聚在这里了：arenas 是内存地盘，central[136] 是 136 个 mcentral，pages 是页分配器，allspans 是 GC sweep 遍历用的 span 全集。central 数组中每个 mcentral 后都有 cache-line padding，防止伪共享。fixalloc 是堆外分配器 —— mspan 和 mcache 自身也需要内存，但不能用 mallocgc（否则死循环），所以用专门的固定大小分配器。

三层之间的”货币”。 理解整个体系的关键之一是看每层之间交换的粒度：

交换路径 交换单位 说明
mcache ↔ mcentral mspan 一次换一整个 span，摊销锁
mcentral → mheap 页（8KB） 从 Arena 划拨连续页创建新 span
mheap → OS MB 级 sysAlloc 至少 1MB，grow 至少 4MB

每向上一层，交换的粒度都更大 —— 因为更高层的操作更贵，所以每次要多拿一些回来。

页分配器：mheap 怎么找空闲页。 当 mcentral 问 mheap 要新 span 时，mheap 需要在 8KB 页的粒度上找到 N 个连续的空闲页。堆可能有几个 GB、几十万个页，逐页扫描太慢了。页分配器用五级基数树 + 位图的混合结构解决这个问题（src/runtime/mpagealloc.go）。

整个方案的基础是位图。最底层的 pallocData 每块管理 512 页 = 4MB，用 8 个 uint64 记录每页是否已分配：

type pallocData struct {
    pallocBits          // [8]uint64 — 1=已分配, 0=空闲
    scavenged pageBits  // [8]uint64 — 1=已归还 OS
}

在一个位图块内部，通过位运算可以快速找到连续的空闲页。但如果堆很大，逐块扫描所有位图仍然不够快 —— 这就需要在位图之上建一层摘要索引。Go 在位图之上建了五级摘要树。每个节点是一个 pallocSum，打包在 64 位整数中，记录三个值：start（区域开头的连续空闲页数）、max（区域内最大的连续空闲段）、end（区域末尾的连续空闲页数），各占 21 bit。

为什么同时需要 start、max 和 end？因为连续空闲页可能跨越子节点的边界——一个子节点末尾的空闲页 + 下一个子节点开头的空闲页，合在一起可能构成更大的连续空间。三个值让父节点能准确计算这种跨边界的情况。

五级结构从根到叶分别覆盖 ~16GiB → ~2GiB → ~256MiB → ~32MiB → 4MiB（对应一个位图块）。搜索时先看高层摘要快速排除大片区域，再深入到具体的位图块定位页。页分配器还维护一个 searchAddr，它只增不减，标记”此地址以下全部已分配”。每次分配都从 searchAddr 开始搜索，跳过已知的满区域，大幅缩减搜索范围。

mallocgc：一切汇聚的入口。 前面所有铺垫到此完成。mallocgc 是 Go 所有堆分配的统一入口（src/runtime/malloc.go），它把前面所有概念串成一条执行路径：

func mallocgc(size uintptr, typ *_type, needzero bool) unsafe.Pointer

mallocgc 根据对象大小走不同路径。零大小对象（size == 0）返回全局哨兵 &zerobase，所有零大小对象共享同一地址。Tiny 对象（< 16B && noscan）走 mcache 的 Tiny 分配器，多个小对象合并到一个 16B block。小对象（≤ 32KB）是最常见的路径：查表得到 size class，算出 span class，从 mcache.alloc[spc] 取出 mspan，用 nextFreeFast 找空闲 slot，99% 在这里完成；如果 span 满了，走 refill → mcentral → mheap 逐级上升。大对象（> 32KB）没有 size class，直接以页为单位从 mheap 分配，由页分配器找连续空闲页。

分配不是孤立的操作，它和 GC 始终保持着动态平衡。GC Assist 要求分配内存前先帮忙做标记工作，分配越多帮忙越多，防止分配速度超过 GC 速度。Allocate-black 让 GC 标记阶段新分配的对象自动标为黑色（存活），不会被当前轮回收。比例 Sweep 在 mheap.alloc 前触发，确保堆增长和回收保持平衡。

回收：GC 与 Sweep

Go 使用并发三色标记-清扫 GC。完整的 GC 机制是另一个大话题，这里聚焦于它和内存分配器的交互 —— 特别是 sweep 阶段怎么把”死掉的对象”变成”可用的空间”。

GC 标记位图本身也需要内存，由专门的 gcBitsArena 管理（src/runtime/mheap.go）。系统维护 free、next、current、previous 四个指针，每个 GC 周期结束时轮转一次：previous 释放进 free 池，current 变成 previous，next 变成 current，next 置空等待下次分配。为什么不直接释放 previous？因为 sweep 完成前，旧的 allocBits（它指向上一轮 gcmarkBits）可能还在被读取，需要多等一轮才能安全回收。

Sweep 做了什么？ 回顾上面的位图交换：sweep 一个 mspan 时，把 gcmarkBits 变成新的 allocBits，分配全新的 gcmarkBits（全零），重置 freeindex 和 allocCount。没有逐对象的释放操作。

sweep 完成后，mspan 根据状态流向不同的地方——有空闲 slot 的回到 mcentral 的 partial 列表继续服务；全满的留在 full 列表等下一轮 GC；全空的则整个归还 mheap，页被回收用于创建新 span。这就是 mspan 生命周期的完整闭环：mheap 创建 → mcentral 分发 → mcache 使用 → GC sweep → 回到 mcentral 或归还 mheap。

sweepgen 状态机。 怎么判断一个 mspan 是否需要 sweep？每个 span 有自己的 sweepgen，通过和全局 mheap_.sweepgen 比较即可：差 2 = 需要 sweep，差 1 = 正在 sweep，相等 = 已完成。每个 GC 周期全局 sweepgen 加 2，无需加锁，一次整数比较就能判断。这和 mcentral 的双缓冲设计配合：partial/full 的两组正是通过 sweepgen 区分 swept 和 unswept。

写在最后

整个内存管理体系可以用一句话串联：mallocgc 根据对象大小查找 size class，从 mcache 中对应的 mspan 上用 CTZ 找空闲 slot；mspan 耗尽时从 mcentral 换一个新的；mcentral 没库存时 mheap 通过页分配器在 Arena 中找连续空闲页创建新 mspan；Arena 不够时向 OS 批量申请。GC sweep 通过位图交换批量回收 mspan 中的死对象，让 mspan 重新回到 mcentral 或 mheap 继续流转。

性能关键设计：

allocCache + CTZ 找空闲 slot 一条 CPU 指令
Tiny 分配器 小 noscan 对象浪费 减少 ~12% 分配次数
Size Class 内存碎片 同尺码完美复用
Span 级 refill 频繁加锁 批量交换摊销开销
基数树 pallocSum 找连续空闲页 O(log N) 搜索
位图交换 sweep 逐对象释放 指针交换即完成
sweepgen 双缓冲 sweep 状态判断 无锁整数比较
zeroedBase 水位线 多余的清零 新页本身已零

[!tip] 阅读建议
从 mallocgc 函数开始，顺着调用链走：mallocgc → mcache.nextFree → mcache.refill → mcentral.cacheSpan → mheap.alloc。遇到不认识的结构体再跳到对应文件，这样每个概念都有上下文，比孤立阅读结构体定义有效得多。
